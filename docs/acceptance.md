# FishMark 验收标准

本文定义 FishMark MVP 的验收基线。评审和人工验收都应以此为准，结论必须明确为 PASS 或 FAIL。

---

## 1. 核心原则

- Markdown 文本是唯一事实来源。
- 保存不能自动重排整个文档。
- 编辑必须保持 round-trip 保真。
- 编辑体验应更接近 WYSIWYM，而不是传统富文本编辑器。
- 光标行为必须稳定、可预测。
- 必须同时支持 macOS 和 Windows。

## 2. 文件操作

MVP 最终必须支持：
- 新建 Markdown 文件
- 打开现有 `.md` 文件
- 将 `.md` 文件拖入应用窗口后打开该文件
  - 空工作区时在当前窗口打开
  - 当前窗口已有文档时新开编辑器窗口打开
- 用 `Ctrl/Cmd + S` 保存
- 另存为新文件
- UTF-8 编码文本

自动保存最终必须：
- 默认开启
- 在停止输入一段时间后保留未保存内容
- 在保存失败时避免数据丢失

崩溃恢复最终必须：
- 在异常退出后恢复最近的未保存编辑状态

## 3. 编辑器行为

编辑器最终需要正确处理 Typora 类工作流中常见的 Markdown 结构，包括：
- 标题
- 段落
- 列表和任务列表
- 引用块
- 行内代码
- 代码块
- 链接
- 图片

当前激活块可以保持源码形式，周边块可以渲染，但用户必须始终能够直接编辑 Markdown。

撤销和重做必须自然；删除 Markdown 语法时，不能出现令人意外的光标跳跃。

## 4. 输入法与光标稳定性

编辑器必须避免：
- IME 组合输入期间丢字
- 输入法输入期间光标意外移动
- 过早解析尚未完成的组合文本
- 在行或块之间发生破坏正常编辑的跳转

## 5. 图片与资源

MVP 最终必须支持图片粘贴和拖放。导入的资源应写入本地存储，并尽量以相对路径 Markdown 插入。

## 6. 大纲、搜索与导出

以下能力属于产品基线，即使由较后的 backlog 任务交付：
- 标题大纲导航
- 全文查找替换
- HTML 导出
- PDF 导出

## 7. 性能

编辑器需要能处理长文档。基线目标是：面对 5000 行以上 Markdown 文档时，仍可正常滚动和编辑，不应出现明显输入卡顿。

## 8. 非 MVP 范围

以下内容不属于 MVP 验收范围：
- 云同步
- 协作编辑
- 账号系统
- 插件市场
- 专有富文档锁定格式

## 9. 评审结论规则

评审结果必须明确：
- 满足要求时记为 PASS
- 存在阻塞性缺口时记为 FAIL

不接受模糊结论。

## 10. 阶段性人工验收使用规则

当前阶段到底哪些能力已经进入人工验收，以 `MVP_BACKLOG.md` 和 `docs/progress.md` 为准。`docs/acceptance.md` 只保留稳定的验收基线与能力清单，不再重复维护“截至某天已完成哪些 TASK”。

使用规则：
- 先在 `MVP_BACKLOG.md` 确认对应 task 的状态和执行切片已同步
- 再在 `docs/progress.md` 确认该 task 的阶段状态已同步
- 只有两者一致时，才按下面对应能力的人工验收步骤执行
- 如果能力已写进本文，但对应 task 还没有在 backlog / progress 中同步到可验收状态，则该能力仍视为“未进入验收”

### 骨架验收前置条件

1. 在仓库根目录安装依赖：`npm ci`
2. 确认本机具备可运行 Electron 的桌面环境
3. 使用 Windows 或 macOS 进行验证

### 当前可人工验收的内容

1. 运行 `npm run dev`
2. 确认 Electron 窗口可以启动
3. 确认页面显示 `FishMark`，并且空态更像桌面编辑器而不是居中 demo 卡片
4. 打开 `File` 菜单，确认存在 `Open...`、`Save`、`Save As...`
5. 确认页面仍能显示 preload 暴露的 `platform` 字段
6. 从 `File > Open...` 选择一个 UTF-8 编码的 `.md` 文件
7. 确认页面显示当前文档名、完整路径和文件文本内容
8. 确认文本被加载到 CodeMirror 编辑区中，并可直接编辑 Markdown 文本
9. 修改文本后确认界面显示 `Unsaved changes`
10. 按 `Ctrl/Cmd + Z` 与 redo 对应快捷键，确认基础 undo / redo 可用且不出现异常跳光标
11. 按 `Ctrl/Cmd + S` 或从 `File > Save` 保存，确认状态切回 `All changes saved`，并确认原文件内容已更新
12. 按 `Shift + Ctrl/Cmd + S` 或从 `File > Save As...` 另存为，选择新路径后确认界面路径切换到新文件，且新文件内容与当前编辑文本一致
13. 关闭并重新启动，确认开发壳仍能正常启动

### 自动保存人工验收

仅在 `TASK-005` 已经在 `MVP_BACKLOG.md` 与 `docs/progress.md` 中同步为已交付状态时执行：

1. 打开一个已有路径的 `.md` 文件
2. 修改文本后停止输入约 1 秒，确认未手动保存时磁盘文件已更新
3. 再次修改文本并让编辑器失焦，确认会立即触发自动保存
4. 确认自动保存期间界面显示 `Autosaving...`
5. 确认自动保存成功后状态回到 `All changes saved`
6. 触发一次保存失败场景，确认界面明确提示“更改仍在内存中”，且文档仍保持未保存状态

### 当前不应通过人工验收的内容

以下能力如果对应 task 还没有在 `MVP_BACKLOG.md` 与 `docs/progress.md` 中同步到可验收状态，应直接判定为“未进入验收”：
- Markdown 编辑与渲染
- 图片导入
- 大纲、搜索替换、导出
- 崩溃恢复
- 中文输入法专项稳定性
