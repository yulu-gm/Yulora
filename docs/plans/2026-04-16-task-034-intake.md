Task: TASK-034
Goal: 在保留现有 top-level `blockMap` 与 active-block 语义的前提下，为 `packages/markdown-engine` 建立可长期扩展的完整 inline AST 基础，并让 `packages/editor-core` 使用这套 AST 驱动非激活态 `bold / italic / inline code / strikethrough` 渲染，为紧接着的链接、图片和复杂嵌套任务留出稳定承接面。

In scope:
- 在 `packages/markdown-engine/` 定义 canonical `MarkdownDocument` 与 inline AST 类型
- 为 `heading`、`paragraph`、`list item`、`blockquote line` 提供 absolute-offset inline AST
- 在当前仓库内补一份本地 `strikethrough` micromark extension，不新增外部依赖
- 保持 `parseBlockMap()` 兼容，同时新增面向后续任务的 `parseMarkdownDocument()` 入口
- 让 `packages/editor-core/` 用 inline AST 生成非激活态行内装饰
- 覆盖常见嵌套，包括 `***text***`、`~~**text**~~`、`**a \`code\` b**`
- 保护 IME / selection / undo-redo / round-trip / CRLF 边界

Out of scope:
- 链接点击、系统浏览器打开与 preload/main bridge
- 图片落盘、拖放导入与资源命名策略
- reference-style link/image definitions 与完整 CommonMark/GFM 全覆盖
- inline HTML、table、数学公式、mermaid 等新的 Markdown 方言能力
- 把当前 top-level block map 改造成整份文档的单棵 block AST

Landing area:
- `packages/markdown-engine/src/`
- `packages/editor-core/src/`
- `src/renderer/styles.css`
- `src/renderer/code-editor.test.ts`
- `docs/test-cases.md`

Acceptance:
- 非激活态 `**`、`*`、`` ` ``、`~~` 显示为渲染文本而不是原始 marker
- 聚焦回对应 block 后，完整 Markdown 源码立即恢复并可直接编辑
- 常见嵌套在非激活态可稳定显示，激活态不丢失原始 Markdown 文本
- `packages/markdown-engine` 能输出稳定的 inline AST，至少覆盖 `text / strong / emphasis / strikethrough / codeSpan / link / image`
- `editor-core` 不重新解析 Markdown，只消费 `markdown-engine` 产出的结构
- 不破坏现有 heading / paragraph / list / blockquote / code fence / thematic break 渲染链路
- 不影响选择、复制、IME 组合输入和撤销重做

Verification:
- `npm run test -- packages/markdown-engine/src/parse-inline-ast.test.ts packages/markdown-engine/src/parse-block-map.test.ts packages/editor-core/src/decorations/block-decorations.test.ts src/renderer/code-editor.test.ts`
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`
- 人工检查：普通段落、列表项、引用块中分别输入嵌套 inline 格式，确认激活 / 非激活切换与 IME 表现稳定

Risks:
- inline AST range 或 marker range 计算错误会直接破坏点击映射、复制和回源码态
- `~~` 本地 extension 若与 attention 规则冲突，容易在复杂嵌套里产出错误树结构
- inline signature 设计不稳会导致 decoration 频繁全量刷新，影响长文档性能
- list item 与 blockquote line 的内容范围若切错，会把 marker 一起样式化或漏掉 continuation text
- 当前仍未引入 viewport 级装饰裁剪，行内装饰数增加后要关注大文档压力
