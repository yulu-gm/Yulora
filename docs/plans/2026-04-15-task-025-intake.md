# TASK-025 Intake

Task: TASK-025
Goal: 建立只面向开发和 agent 的独立测试工作台窗口，并提供测试模式启动入口；工作台负责显示 debug 信息和测试进程状态，具体测试再拉起主编辑器窗口。
In scope:
- 在 `src/main/` 中增加测试模式判定和独立 workbench 窗口创建逻辑
- 在 `src/preload/` 中暴露最小测试模式 bridge
- 在 `src/renderer/` 中增加基础测试工作台页壳
- 保证测试模式不会污染正常主编辑器启动和窗口生命周期
Out of scope:
- 场景注册表与场景元数据模型
- 测试运行器与步骤状态机
- CLI 统一入口
- 真正的 debug 流、结果工件和 visual-test 能力
Landing area:
- `src/main/main.ts`
- `src/main/paths.ts`
- `src/main/*.test.ts`
- `src/preload/preload.ts`
- `src/renderer/App.tsx`
- `src/renderer/main.tsx`
- `src/renderer/styles.css`
- `src/renderer/types.d.ts`
- `src/renderer/*.test.ts`
Acceptance:
- 可以独立打开测试工作台窗口
- 测试工作台与主编辑器窗口隔离
- 可在测试模式下启动，且不影响正常开发壳
- 测试工作台能展示基础场景列表页壳、debug 区和测试进程区
- 测试工作台提供后续拉起主编辑器测试窗口的入口占位
Verification:
- `npm run test -- src/main/paths.test.ts src/main/package-scripts.test.ts`
- `npm run test -- src/main/test-workbench*.test.ts`
- `npm run test -- src/renderer/test-workbench*.test.ts`
- `npm run lint`
- `npm run typecheck`
- `npm run build`
Risks:
- Electron 窗口生命周期分叉，容易误伤正常 `activate` / `window-all-closed` 流程
- preload bridge 如果泄漏过多能力，会破坏 main/preload/renderer 边界
- renderer 条件分支如果直接侵入主编辑器页面，后续扩展会变脆
