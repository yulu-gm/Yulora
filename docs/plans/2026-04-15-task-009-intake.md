# TASK-009 Intake

Task: `TASK-009`
Goal: Let the editor identify the current top-level Markdown block from the cursor position and keep that active-block state in sync as selection moves, without introducing visible block rendering changes yet.

In scope:
- define the minimal active-block state model under `packages/editor-core/`
- map CodeMirror selection changes to the current top-level block from `parseBlockMap()`
- expose a renderer-consumable query/update surface for current block information
- keep the current document/save/autosave flow intact
- add focused tests for block resolution and selection-driven updates

Out of scope:
- visible active-block UI or debug badges
- heading/paragraph/list/blockquote rendering behaviors from `TASK-010` to `TASK-013`
- block decorations, widgets, or source/render mode toggles
- nested block semantics beyond the existing top-level block map
- new file/menu/autosave features

Landing area:
- `packages/editor-core/src/active-block.ts`

- `packages/editor-core/src/active-block.test.ts`

- `src/renderer/code-editor.test.ts`
- `src/renderer/code-editor-view.tsx`
- `src/renderer/App.tsx`

- `docs/decision-log.md`

Acceptance:
- current cursor position resolves to the expected top-level block when one exists
- active block updates as the selection anchor moves between blocks
- renderer can consume current block info without reparsing in React state code
- typing, undo/redo, blur autosave, and manual save behavior stay intact

Verification:
- `npm run test -- packages/editor-core/src/active-block.test.ts`
- `npm run test -- src/renderer/code-editor.test.ts src/renderer/app.autosave.test.ts`
- `npm run lint`
- `npm run typecheck`
- `npm run test`
- `npm run build`

Risks:
- cursor-to-block boundary handling can go off by one around blank lines and block ends
- reparsing on every selection update must not interfere with input, IME stability, or undo/redo semantics
- renderer wiring must not couple future block rendering tasks to transient UI state too early
