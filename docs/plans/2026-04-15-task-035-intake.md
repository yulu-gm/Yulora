# TASK-035 Intake

Task: `TASK-035`
Goal: Establish an IME-safe editing baseline before block-rendering work starts, so future decoration or source/render switching does not cause composition text loss or cursor jumps in common Chinese input flows.

In scope:
- reproduce the current IME-sensitive update path around CodeMirror document and selection updates
- add a composition-session guard that keeps the editor on the stable plain-text path while IME input is active
- cover paragraph, heading, and list composition flows with focused regression tests
- record current limitations and non-covered IME scenarios for follow-up work before `TASK-022`

Out of scope:
- visible block rendering, decorations, or source/render switching from `TASK-010` to `TASK-013`
- a full Chinese IME compatibility pass across all Markdown constructs
- quote/code block specific IME handling unless needed to keep the baseline architecture coherent
- changes to save, autosave, preload, or main-process file flows
- Playwright or end-to-end automation for IME in this round

Landing area:
- `src/renderer/code-editor.ts`

Acceptance:
- Chinese IME composition in plain paragraph, heading, and list input does not lose committed text
- the cursor/selection does not jump unexpectedly during composition in those covered scenarios
- the editor exposes a clear guard point that later block-rendering tasks can reuse to defer disruptive view updates during composition
- known uncovered IME limits are documented explicitly

Risks:
- CodeMirror composition behavior is sensitive to transaction timing, selection remapping, and DOM event ordering
- a guard that is too broad could hide legitimate selection or active-block updates after composition commits
- later block-rendering tasks may need a reusable contract instead of one-off renderer conditionals
- IME behavior can differ between Windows and macOS, so this round should avoid overfitting to one platform-specific event sequence
