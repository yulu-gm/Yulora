# TASK-039 Intake

Task: `TASK-039`
Goal: Add the first stable thematic break rendering path so non-active separators read like rendered horizontal rules, while active separators remain raw Markdown and preserve the original `---` or `+++` source text.

In scope:
- extend `packages/markdown-engine` block-map types with top-level `thematicBreak` support
- recognize `---` through `micromark` thematic break events and explicitly support `+++` as a Yulora markdown-dialect extension
- preserve exact source offsets and line ranges for separator blocks
- render non-active separators through the existing `src/renderer/code-editor.ts` decoration pipeline
- restore full Markdown source as soon as the caret moves back onto the separator line
- add regression coverage for parser output, inactive/active rendering transitions, and CRLF document replacement

Out of scope:
- `_ _ _` or `***` thematic break support
- nested separator semantics inside other containers
- dedicated input shortcuts for inserting separators
- changes to `main`, `preload`, save bridges, autosave policy, or React shell structure

Acceptance:
- `parseBlockMap()` emits top-level `thematicBreak` blocks for `---` and `+++` with exact offsets and line ranges
- moving the cursor outside a separator line renders it as a non-active horizontal rule
- moving the cursor back onto the separator restores the raw Markdown source immediately
- replacing the loaded document with CRLF content keeps separator decorations aligned
- existing block rendering and composition-guard behavior remain intact

Risks:
- `+++` is not a CommonMark thematic break, so the extension must stay narrow and deterministic to avoid surprising future markdown features
- line-level decorations for separators must not distort cursor mapping or interact badly with neighboring blank lines
- parser offsets must continue to match CodeMirror-normalized document text, especially for CRLF inputs
