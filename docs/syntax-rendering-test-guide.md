# 语法渲染测试指南

日期：2026-04-16

---

## 1. 目的

本文总结 FishMark 在实现 fenced code block 渲染时暴露出的测试经验，作为后续所有 Markdown 语法渲染能力的统一测试指南。

适用范围包括但不限于：
- 标题
- 引用块
- 列表 / 任务列表
- 代码块
- 表格
- 数学公式
- mermaid
- 其他未来新增的块级或行内渲染语法

核心原则：
- 渲染只是视图层能力，不是数据真相
- Markdown 源文本必须始终可往返还原
- 测试不能只验证“渲染出来了”，必须验证“边界交互仍然稳定”

## 2. 这类功能最容易出问题的地方

语法渲染的主要风险通常不在“正常显示”本身，而在下面这些边界：

- 非激活态和源码态之间的切换是否完整
- 光标是否真的落在用户以为的位置
- 点击块顶部、底部、前后空行时是否出现半切换状态
- `Enter`、`Backspace`、`Tab`、`Shift+Tab` 是否破坏 Markdown 结构
- 连续两次或三次操作后，状态机是否仍然一致

经验上，真正的 bug 往往来自“视觉位置”和“真实 selection offset / block offset”不一致。

## 3. 测试分层

每一种新的语法渲染，都应至少覆盖下面四层测试。

### 3.1 结构解析测试

先验证 parser / block map，而不是先看 UI。

至少要断言：
- block type 识别正确
- `startOffset` / `endOffset` 精确
- `startLine` / `endLine` 正确
- 从 source 中按 offset slice 出来的文本与原始 Markdown 完全一致
- 相邻段落、列表、引用块不会串到当前 block

推荐场景：
- 最小合法输入
- 带可选参数的输入，例如 code fence info string
- 与前后普通段落相邻
- 位于文档开头
- 位于文档结尾
- 中间带空行

### 3.2 渲染态切换测试

要验证语法块在非激活态和激活态之间是否能完整切换。

至少要覆盖：
- 光标移出当前块后，非激活态渲染生效
- 光标移回块内部后，完整 Markdown 源码恢复
- 恢复时不是只恢复一部分，而是整个块一起恢复

不要只测块中间位置。块的第一行、最后一行、closing marker 对应行都要测。

### 3.3 边界交互测试

每个语法块都应该显式验证边界交互，而不是顺带验证。

建议固定覆盖：
- 点击块顶部附近
- 点击块底部附近
- 光标落在块前一行
- 光标落在块后一行或空白分隔行
- 在边界位置按 `Enter`
- 在边界位置按 `Backspace`
- 如果语法涉及缩进或层级，再测 `Tab` / `Shift+Tab`

目标是确认：
- 不会出现半渲染半源码状态
- 不会只露出语法标记的一部分
- 不会误删隐藏的 marker / fence
- 不会发生不符合直觉的跳光标

### 3.4 连续操作回归测试

单步测试不够，必须补至少一条连续操作用例。

推荐模式：
1. 进入渲染态
2. 把光标移到块边界
3. 执行一次按键或点击
4. 再执行第二次按键
5. 再检查文档内容、selection 和 decoration

原因很简单：
- 第一次操作常常只是“看起来对了”
- 第二次操作才会暴露 selection 落点错误、状态没收干净、隐藏标记被误删等问题

## 4. 每种语法的最小测试清单

后续新增任何语法渲染时，至少应检查下面这份清单。

### 4.1 Parser / Block Map

- 能识别该语法块
- block 边界精确
- round-trip slice 正确
- 与前后其他 block 不串块

### 4.2 Decoration / Rendering

- 非激活态样式正确
- 激活态完整恢复源码
- 同一文档中与其他已支持 block 并存时不互相污染

### 4.3 Interaction

- 顶部点击稳定
- 底部点击稳定
- 前后空行切换稳定
- `Enter` 行为稳定
- `Backspace` 行为稳定
- 如有必要，`Tab` / `Shift+Tab` 稳定

### 4.4 Multi-step Regression

- 同一路径连续操作两次后仍正确
- 连续操作后文档文本仍合法
- 连续操作后不会留下“半激活”装饰状态

## 5. 编写测试时的具体建议

### 5.1 优先断言真实文档内容

UI 看起来对，不代表文本真的对。

自动化测试里优先断言：
- `getContent()`
- selection anchor / head
- block offset
- 关键 decoration class 是否存在或消失

不要只断言某个 DOM 节点“看起来还在”。

### 5.2 明确区分“状态切换”与“内容修改”

有些按键的第一次触发，预期应该只是把块从渲染态切回源码态，而不是立即修改 Markdown。

比如这次代码块的经验：
- 第一次 `Backspace` 应先回源码态
- 第二次 `Backspace` 才应继续修改代码内容

测试里要把这两步拆开分别验证。

### 5.3 优先测试文档末尾和空行场景

下面这些情况最容易出问题：
- block 在 EOF 结尾
- block 后刚好有一个空行
- block 前后有多个空行
- 光标在最后一个换行后

这些场景必须单独补测试，不要默认“正常情况覆盖到了”。

### 5.4 不要只验证 happy path

每个语法块除了“正常显示”，至少还应验证：
- 与相邻 block 切换
- 从边界位置恢复源码
- 连续删除
- 连续回车
- 重复进入 / 退出渲染态

## 6. 手工验收建议

自动化测试能锁住文本、selection 和 decoration；手工验收则更适合确认真实编辑体感。

每种语法渲染建议在 `docs/test-cases.md` 中补一条用例，至少写清楚：
- 如何进入渲染态
- 如何回到源码态
- 顶部点击、底部点击应怎样表现
- `Enter` / `Backspace` 是否有特殊语义
- 连续两次操作后的预期是什么

手工验收重点关注：
- 光标是否自然
- 是否有闪烁或跳变
- 是否出现用户能看到的半切换状态
- 行为是否符合 Typora 式单栏编辑直觉

## 7. 推荐测试模板

后续新增一种语法渲染时，可以按这个顺序补测试：

1. parser 测试：确认 block 类型和边界
2. inactive rendering 测试：确认非激活态装饰
3. active restore 测试：确认回源码态
4. top boundary 测试：点击或选择顶部边界
5. bottom boundary 测试：点击或选择底部边界
6. key interaction 测试：`Enter` / `Backspace`
7. chained interaction 测试：连续两次操作
8. manual test case：补到 `docs/test-cases.md`

## 8. 判断测试是否足够的标准

如果一项语法渲染只验证了“显示正常”，那还不算测试完成。

只有同时满足下面几点，才算覆盖到位：
- 结构解析可验证
- 渲染态与源码态切换可验证
- 边界交互可验证
- 连续操作可验证
- 手工验收路径已记录

## 9. 一句话结论

语法渲染本质上不是样式功能，而是编辑状态机功能。

因此后续做任何新的语法渲染，都要把测试重点放在：
- block 边界
- selection 落点
- 渲染态 / 源码态切换
- 边界按键
- 连续交互回归

只有这样，才能避免“看起来渲染对了，但编辑一碰就坏”的问题。
